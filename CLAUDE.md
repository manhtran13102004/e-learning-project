## Yêu cầu cốt lõi của dự án (E-learning Project)

Dự án này là một sản phẩm thực tập yêu cầu sự chặt chẽ và tuân thủ các quy tắc kiến trúc sau:

1. **Phân công vai trò:** AI chỉ tập trung hoàn toàn vào việc xây dựng giao diện người dùng (UI - Frontend bằng React).
2. **Kiến trúc hệ thống:** Hệ thống tổng thể sử dụng kiến trúc **Microservices**.
3. **Backend:** Backend được viết bằng **Java** và sử dụng framework **Spring Boot**.
4. **Tích hợp:** UI phải được thiết kế dạng decoupled (tách rời), chuẩn bị sẵn các service layer rõ ràng để dễ dàng tích hợp với các API Gateway và các endpoints của Spring Boot sau này.
5. **Mục tiêu:** Mọi quyết định thiết kế Frontend phải tạo điều kiện để tận dụng tối đa hệ sinh thái của Spring ở dưới Backend.
6. **Dữ liệu giả lập (Mock Data):** Mọi hình ảnh, text, hay thông tin liên quan đến các đối tượng (entity) trong quá trình thiết kế UI hiện tại chỉ mang tính chất demo (placeholder). Các thành phần này phải được thiết kế dưới dạng biến hoặc props linh hoạt để sau này thay thế hoàn toàn bằng dữ liệu thật thông qua API gọi từ backend Spring Boot.
7. **Xóa file/thư mục:** Luôn phải hỏi ý kiến người dùng trước khi xóa bất kỳ file hoặc thư mục nào, không tự ý xóa.
8. **Không đụng vào code backend, nhưng được sửa cấu hình:** Không bao giờ được sửa, tạo, hay thay đổi **code** (file `.java`: entity, controller, service, DTO, mapper, repository, specification...) trong thư mục `zzz/` hay `backend/` (backend), **trừ khi người dùng yêu cầu trực tiếp**. Người dùng muốn tự code logic backend bằng thực lực của mình. Nếu người dùng cần hỗ trợ về backend, mặc định chỉ được gợi ý và hướng dẫn (giải thích, chỉ ra hướng đi, review code), không tự ý viết hay sửa code thay họ trừ khi được yêu cầu rõ ràng.
   - **Ngoại lệ — file cấu hình được phép sửa tự do** (không cần chờ yêu cầu trực tiếp mỗi lần): `application.properties`/`application*.yml`, `pom.xml` (dependency, plugin, version...), `docker-compose*.yml`, `.env`, file log config (`logback*.xml`), và các file cấu hình build/tooling tương tự — miễn là thay đổi phục vụ đúng mục đích cấu hình (kết nối DB, port, dependency, logging...), không chèn logic nghiệp vụ vào đó.
9. **Cấu trúc backend (từ 2026-07-15):** Backend chính thức được code theo kiến trúc **Microservices thật sự** trong thư mục `backend/` (gồm các service riêng biệt như `api-gateway-master`, `authn-service-master`, `author-service-master`, `base-service-master`...). Thư mục `zzz/` là monolith cũ, **giờ chỉ để tham khảo** (không phải nơi phát triển chính thức nữa) — không đề xuất thay đổi coi `zzz/` là nguồn API thật khi trả lời các câu hỏi về backend, trừ khi người dùng chủ động nhắc tới `zzz/`.
10. **Cấu trúc thư mục Frontend:** Toàn bộ code Frontend (React/Vite) nằm trong thư mục `FE/` (bao gồm `src/`, `public/`, `index.html`, `package.json`, các file cấu hình `vite.config.ts`/`tsconfig*.json`/`tailwind.config.js`/`postcss.config.js`/`components.json`/`.oxlintrc.json`/`.env`). Khi chạy lệnh npm/vite hoặc tham chiếu đường dẫn frontend, luôn thao tác bên trong `FE/`.
